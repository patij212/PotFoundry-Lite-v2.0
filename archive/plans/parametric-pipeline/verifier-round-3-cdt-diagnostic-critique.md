### C2 [CRITICAL]: P2 (Remove Filter) — ACCEPT WITH AMENDMENTS

**Generator's claim**: "Remove the filter entirely. Pass ALL 6,586 chain edges as constraints to CDT."

**Risk the Generator identified**: Crossing constraints. The Generator argues the risk is "negligible for adjacent-row interpolated edges" because they're "short, nearly-vertical edges that cannot cross each other."

**Verification**: The interpolated vertices are created at OWT L348-362. They're generated by linear interpolation between two feature vertices across multi-row gaps:
```typescript
const interpU = p0.u + du * frac;  // linear interpolation
const interpRow = p0.rowIdx + dir * s;
```

For a SINGLE chain, consecutive interpolated edges between rows N and N+1 can't cross each other — they connect consecutive interpolated points along a linear path.

**But**: Two DIFFERENT chains can cross. If chain A has U=0.08 at row 14 and U=0.09 at row 15, and chain B has U=0.09 at row 14 and U=0.08 at row 15, their edges cross. This happens when two same-kind features swap positions between rows (e.g., two ridges approaching each other and crossing).

**Required amendment**: Before passing interpolated edges to CDT, check for pairwise crossing within each strip. If two constraint edges cross, keep the one from the longer chain and drop the other. This is O(n²) per strip but n is small (typically <10 chain edges per strip).

**Alternative (simpler, recommended)**: Use a `try/catch` around the CDT call. If CDT throws (crossing constraints detected), fall back to sweep-repair for that strip. The current code already has sweep fallback infrastructure. The Generator's diagnostic shows 0 CDT failures on the primary edges, so crossings in interpolated edges should be rare. When they occur, sweep-repair handles the strip without the crossing constraint.

### C3 [CRITICAL]: P3 (Reduce Companion Density) — REJECT AS STATED

**Generator's claim**: Reduce density from 12 → 2 to fix 87.9% aspect violations.

**User's stated goal**: "I don't think that removing or lowering the companion density is a good idea. I do want the high definition around the feature edges. We can optimize the way we place the points tho."

**The real problem**: Companions are placed **only along the U-axis** (same T-row as the chain vertex). This creates a 1D line of points, not a 2D point cloud. The aspect ratio problem is not from TOO MANY companions — it's from companions being packed densely in U but spaced by the full T-row gap vertically.

Verified at OWT L445-453:
```typescript
companionVertices.push({
    u: cu,
    rowIdx: cv.rowIdx,  // ← SAME ROW as chain vertex
    vertexIdx: nextVertexIdx++,
    chainId: cv.chainId,
    pointIdx: -1,
});
```

**The math** at the default density=4:
- Grid cell width (U): ~0.00173
- Companion spacing in U: cellWidth × gap_fraction × 0.2 ≈ 0.000173
- Row spacing in T: ~0.0032
- Aspect ratio: 0.0032 / 0.000173 = **18.5:1**

At density=12 (the diagnostic run):
- Companion spacing in U: ~0.000073
- Aspect ratio: 0.0032 / 0.000073 = **43.8:1**

**NOTE**: The Generator's diagnostic was run at density=12, but the **default** is density=4 (PEC L433: `pc?.chainStripDensity ?? 4`). The 130K companion count and 43:1 aspect ratio are from a non-default configuration. At the default density=4, there would be ~52K companions with ~18:1 aspect ratio. Still bad, but the Generator's numbers overstate the severity of the default configuration.

### C4 [WARNING]: Generator's Aspect Ratio Analysis Misses the Root Cause

The Generator says: "CDT optimizes Delaunay angles, but the Delaunay criterion is a property of the triangulation, not the vertex set."

This is correct but incomplete. The reason CDT can't help is specifically that companions form a **1D line** (varying U, constant T). Any triangulation of a 1D point set connected to distant points in the other dimension produces slivers. The fix is not fewer points — it's **2D** companion placement.

**The right fix**: Make companion spacing **aspect-aware**. Instead of packing N companions at equal U-fractions within a single row, compute how many companions the local aspect ratio can support:

```
maxCompanionsPerSide = floor(gap / (T_spacing * targetAspect))
```

Where `targetAspect` is the desired max aspect ratio (e.g., 4:1) and `T_spacing` is the distance to the adjacent row.

For a cell gap of 0.000865 (half of 0.00173), T_spacing = 0.0032, targetAspect = 4:
```
maxCompanionsPerSide = floor(0.000865 / (0.0032 / 4)) = floor(0.000865 / 0.0008) = 1
```

This would place **1 companion per side** — which is density=1 anyway. But this reveals something important: at the current grid resolution, ANY companion density >1 creates aspect violations. You can't fix aspects by reducing density from 12 to 2 — even density=2 produces 10:1 aspects.

**The actual solution (two options)**:

**Option A: Aspect-capped companion count** (simple, recommended for now)
- Compute `maxDensity = floor(gap / (tSpacing * TARGET_ASPECT))` per chain vertex
- Use `min(configuredDensity, maxDensity)` as the local companion count
- This caps companions per side to what the local geometry can support without slivers
- High-density areas (where T-spacing is small or U-gap is large) get more companions
- Tight areas get fewer — but they don't NEED more because the grid already has adequate U-resolution there

**Option B: 2D companion cloud** (better quality, more complex)
- For each chain vertex, place companions in a 2D disc pattern, not just a U-line
- Disc radius = half the grid cell width
- Point spacing within disc = T_spacing / targetAspect
- This produces isotropic vertex density around chain points
- BUT: companions at different T-positions need their own R-values (surface evaluation), which currently only exists for grid rows. This would require interpolating R between adjacent rows or probing the GPU.

**Recommendation**: Start with Option A. It's a ~10-line change to the existing companion loop that respects the user's desire for high definition while eliminating the worst slivers. Option B is v3.0 work.
