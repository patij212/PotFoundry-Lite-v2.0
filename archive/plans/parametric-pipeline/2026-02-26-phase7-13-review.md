## Issues Found

### I1 (Minor): Hardcoded seam thresholds inconsistency

Three different seam thresholds used:
- `SEAM_ZONE = 0.15` in `seamSafeMidpointU()` (AdaptiveRefinement.ts:291)
- `SEAM_THRESHOLD = 0.02` in `globalSmoothing()` (AdaptiveRefinement.ts:1804)
- `seamThreshold = 0.02` default in `identifySeamVertices()` (EdgeCollapser.ts:391)

The SEAM_ZONE=0.15 is semantically different (it's a guard for when to use circular vs linear midpoint), but the 0.02 values should be unified into a shared constant.

**Severity:** Minor. No functional impact — both serve correct purposes.

### I2 (Minor): `globalSmoothing()` blends U without seam safety

Line 1884: `const blendU = (oldU + newU) * 0.5` — the blend between old and smoothed U is a linear average, not a circular blend. If `oldU ≈ 0.01` and `newU ≈ 0.98`, the blend would be `0.495` instead of `≈ 0.0`.

However, seam vertices are already excluded from smoothing (line 1814), so this path is never hit for seam vertices. The risk is vertices at `u = 0.15 + ε` which are NOT in the seam exclusion zone but might have a neighbor at `u = 0.85` — edge case, unlikely on typical pot meshes.

**Severity:** Minor. Seam exclusion protects against the worst case.

### I3 (Info): `_evaluateQuadric` is unused

`EdgeCollapser.ts:156–177`: The standalone `_evaluateQuadric()` function is defined but never called. `computeCollapseCost()` inlines the quadric evaluation. The underscore prefix suggests this was intentional (kept as documentation/reference).

**Severity:** Info. Lint may flag as unused. Consider removing or annotating with `// eslint-disable-next-line`.

### I4 (Info): Duplicate 1-ring function

`EdgeCollapser.ts` has both `oneRing()` (lines 239–250, for `Map<number, number[]>`) and `oneRingSet()` (lines 273–284, for `Map<number, Set<number>>`). Only `oneRingSet` is used by the main collapse path. `oneRing` is used only by the exported `checkLinkCondition` (for tests).

**Severity:** Info. Consider consolidating or documenting the dual API.

### I5 (Minor): `globalSmoothing` area weighting double-counts neighbors

In `globalSmoothing()` (lines 1863–1873), for each triangle incident on vertex `v`, all non-`v` vertices of that triangle are weighted by the triangle's area. But the same neighbor vertex `w` may appear in multiple incident triangles. Each occurrence adds `area_t` weight. This means the weight for `w` is the sum of areas of triangles containing both `v` and `w` — which is actually correct for cotangent Laplacian behavior. The implementation is fine; just noting it's area-of-shared-triangles weighting, not per-neighbor area weighting.

**Severity:** Info. Behavior is mathematically defensible.

### I6 (Minor): Aspect ratio in `computeMeshQuality` uses edge ratio, not standard definition

The standard aspect ratio for triangles is `longest_edge / (2 * inradius)` or `circumradius / inradius`. The current implementation (lines 1756–1765) uses `maxEdge / minEdge`, which is the edge length ratio. For equilateral triangles both give 1.0, but for slivers they diverge. The edge ratio is a valid quality metric but is not the standard "aspect ratio" used in mesh processing literature.

**Severity:** Minor. Consistent with the existing codebase convention. May cause confusion if comparing with external tools.
