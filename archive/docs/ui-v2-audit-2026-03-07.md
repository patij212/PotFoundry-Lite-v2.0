UI v2 ships with **two P0-level issues** (one crash vector, one broken feature promise), **five P1
behavioral failures** (silent errors, accessibility violation, wrong documentation), and **at least
four P2 quality regressions** compared to v1. The progressive-disclosure architecture, accessibility

**All issues have confirmed fix paths.** Total estimated change: ~120 LOC across 9 files. The fixes
are organized in three phases (quick wins → component refactors → store schema change), enabling a

#### P0-A · Missing ErrorBoundary on AppUIv2
**Status**: CONFIRMED by all agents  
**File**: [`src/App.tsx`](../src/App.tsx) lines 537–539  
**Blast radius**: Any unhandled React render error in any of the 20 v2 components → entire v2 UI
disappears to blank screen with no recovery path except full page reload.

**Evidence**:
```tsx
// App.tsx L537–539 — NO ErrorBoundary
<Suspense fallback={null}>
    <AppUIv2 />
</Suspense>
```
v1 `AppUI.tsx` has three named boundaries at lines 91, 96, 103, 106.
`src/ui/shared/ErrorBoundary.tsx` is a standard class component with `getDerivedStateFromError`
and `componentDidCatch`. It is exported from `src/ui/shared/index.ts`. The prop API is
`<ErrorBoundary name="string">`.

**Note** (Verifier): The class boundary catches synchronous render errors only. Async errors from
WebGPU Promise chains and event handlers are uncovered. A `window.addEventListener('unhandledrejection')`
in `main.tsx` provides the async safety net.

**Fix**:
```tsx
// App.tsx — wrap AppUIv2
import { ErrorBoundary } from './ui/shared';

// ...
<ErrorBoundary name="AppUIv2">
    <Suspense fallback={null}>
        <AppUIv2 />
    </Suspense>
</ErrorBoundary>
```
```ts
// main.tsx — add after React root creation
window.addEventListener('unhandledrejection', (event) => {
    console.error('[PotFoundry] Unhandled promise rejection:', event.reason);
});
```

**Executioner verdict**: FEASIBLE. `main.tsx` is 24 lines; the addition is safe. No import
conflicts. Phase 2 of the implementation order.

#### P0-B · Export format selector is dead UI — "3MF" selection has zero effect
**Status**: CONFIRMED by all agents (with CRITICAL AMENDMENT: 3MF exporter exists)  
**Files**: [`src/ui/v2/tabs/ExportTab.tsx`](../src/ui/v2/tabs/ExportTab.tsx) L125;
[`src/ui/v2/layout/StatusFooter.tsx`](../src/ui/v2/layout/StatusFooter.tsx) L79;
[`src/hooks/useExport.ts`](../src/hooks/useExport.ts) L307

**Blast radius**: Intermediate users (confidence level ≥ 2) who select "3MF" always download an
STL file. The 3MF button is a false promise. The 3MF exporter infrastructure (`export3MF.ts`,
`stlExport.ts` routing) already exists and is fully operational.

**Evidence**:
```tsx
// ExportTab.tsx L125 — local state, never leaves component
const [format, setFormat] = useState<ExportFormat>('stl');

// StatusFooter.tsx L79 — always STL
await exportSTL();    // ignores format entirely

// stlExport.ts (already built) — routing IS there
if (format === '3mf') { const { exportTo3MF } = await import('./exporters/export3MF') }
```

**Format selector visibility**: gated by `isVisible('export:format')` at confidence level ≥ 2.
New users are not yet affected, but intermediate users are actively misled.

**Fix** (5-step plumbing, no new code):
1. Add to `src/state/slices/ui.ts`:
   ```ts
   exportFormat: 'stl' | '3mf';   // default: 'stl'
   // + setExportFormat action
   ```
   Declare the `'stl' | '3mf'` type inline — do NOT import `ExportFormat` from `stlExport.ts`
   (circular import risk).
2. `ExportTab.tsx`: replace `useState('stl')` with `useAppStore` selector for `exportFormat`.
3. `useExport.ts` `exportSTL()`: add `format?: 'stl' | '3mf'` param; use `downloadMesh()` from
   `stlExport.ts` which already dispatches by format.
4. `StatusFooter.tsx` `handleDownload`: read `exportFormat` from store, pass to `exportSTL(format)`.
5. Confirm `ui` slice is not in `PERSISTED_KEYS` (correct) — format resets to `'stl'` on page
   reload (acceptable default).

**Executioner verdict**: FEASIBLE WITH NOTES. This is a 5-file store schema change — run
`tsc --noEmit` before merging. Phase 3 of implementation order.

#### P1-C · WelcomeCard accessibility violation — role="complementary" blocks screen readers
**Status**: CONFIRMED (Verifier: CSS cascade risk is mitigated)  
**File**: [`src/ui/v2/onboarding/WelcomeCard.tsx`](../src/ui/v2/onboarding/WelcomeCard.tsx) line 66

**Evidence**:
```tsx
<div
    className={`pf2-welcome${exiting ? ' pf2-welcome-exit' : ''}`}
    role="complementary"         // ← WCAG violation: visually modal, semantically a landmark
    aria-label="Welcome to PotFoundry"
>
```
Screen readers can navigate "behind" the WelcomeCard to all interactive content. The WelcomeCard
is visually modal but lacks `role="dialog"` + `aria-modal="true"` + a focus trap. Violates
WCAG 2.1 SC 4.1.2.

**Accessibility fix** (minimum viable, avoids Radix migration risk to CSS animations):
```tsx
<div
    role="dialog"
    aria-modal="true"
    aria-labelledby="pf2-welcome-title"
    className={`pf2-welcome${exiting ? ' pf2-welcome-exit' : ''}`}
>
    <h1 id="pf2-welcome-title">…</h1>
```
Add basic focus cycling between the two action buttons via `onKeyDown` Tab trap (no new
dependencies needed).

**Note on Radix Dialog**: Radix Dialog portals render into `document.body`. `AppUIv2.tsx` already
syncs theme to `document.documentElement.dataset.theme`, so CSS custom properties inherit correctly
through portals. CSS cascade risk is **not a blocker** if Radix migration is preferred in future.
The minimum viable fix above avoids the exit animation conflict.

**Executioner verdict**: FEASIBLE WITH NOTES. Phase 2, standalone commit.

#### P1-E · useConfidence silent persistence failure in private/incognito mode
**Status**: CONFIRMED (Verifier: Generator's `initializedRef` claim was fabricated — target the
actual `saveState()` catch block)  
**File**: [`src/ui/v2/onboarding/useConfidence.ts`](../src/ui/v2/onboarding/useConfidence.ts) ~L108

**Evidence**:
```ts
} catch {
    // localStorage quota exceeded or blocked — silently fail
}
```
In Safari Private and Firefox Strict mode, `localStorage.setItem` throws. Confidence level is
never persisted. On every page reload in incognito, level defaults to 0 — WelcomeCard re-appears
and the FourierBloom preset override re-runs, destroying the user's chosen style.

**Fix** (minimum viable — maintain the correct silent-catch approach):
```ts
let storageAvailable = true;

function saveState(): void {
    if (!storageAvailable) return;
    try {
        localStorage.setItem(STORAGE_KEY, JSON.stringify({ ... }));
    } catch {
        storageAvailable = false;
        if (import.meta.env.DEV) {
            console.info('[useConfidence] localStorage unavailable — running in ephemeral mode');
        }
    }
}
```
This prevents repeated failed write attempts and documents the degraded mode in dev tools.

**Note**: Moving `useConfidence` state to Zustand persist middleware would eliminate this class
of issue entirely, but that is a larger refactor outside the scope of this audit.

**Executioner verdict**: (deferred to Phase 1 lightweight fix as above; the module-level flag
approach is clean and non-breaking).

#### P2-C · WelcomeCard exitTimeoutRef not cleared on unmount
**Status**: CONFIRMED by all agents  
**File**: [`src/ui/v2/onboarding/WelcomeCard.tsx`](../src/ui/v2/onboarding/WelcomeCard.tsx) line 65

**Evidence**:
```tsx
const exitTimeoutRef = useRef<NodeJS.Timeout | null>(null);
// handleExit: exitTimeoutRef.current = setTimeout(() => { setDismissed(true); unlock(…); }, 220);
// NO cleanup useEffect exists for this ref
```
If confidence is unlocked externally mid-animation, WelcomeCard unmounts with a live 220ms timer.
When it fires: `setDismissed(true)` runs on unmounted component, `unlock()` double-fires.

**Fix** (4 lines):
```tsx
useEffect(() => {
    return () => {
        if (exitTimeoutRef.current) clearTimeout(exitTimeoutRef.current);
    };
}, []);
```

**Executioner verdict**: FEASIBLE (trivial). Phase 1, same commit as P1-D.

#### ADD-1 · SidebarV2 resize stuck if mouse released outside browser window
**Status**: CONFIRMED (Inferred-2 from Generator, confirmed by Verifier and Executioner)  
**File**: [`src/ui/v2/layout/SidebarV2.tsx`](../src/ui/v2/layout/SidebarV2.tsx) lines 87–105

**Evidence**:
```ts
document.addEventListener('mouseup', handleResizeEnd);
// mouseup does NOT fire if button released outside browser window
```
`isResizing` stays `true` permanently; sidebar continues tracking pointer movement; cursor
stays `ew-resize`.

**Fix**:
```ts
window.addEventListener('blur', handleResizeEnd);
// + matching removeEventListener in cleanup
```
`window.blur` fires on tab/window focus loss. Double-fire on simultaneous mouseup+blur is harmless
(`setIsResizing(false)` is idempotent).

**Executioner verdict**: FEASIBLE. Phase 1.

#### ADD-2 · WelcomeCard Escape key listener has stale `handleExit` closure
**Status**: CONFIRMED (Omission-3 from Verifier)  
**File**: [`src/ui/v2/onboarding/WelcomeCard.tsx`](../src/ui/v2/onboarding/WelcomeCard.tsx) ~L47

**Evidence**:
```tsx
useEffect(() => {
    // ...
    window.addEventListener('keydown', handleKeyDown);
    return () => window.removeEventListener('keydown', handleKeyDown);
}, [level, dismissed]);   // ← handleExit missing from deps
```
`handleExit` (which reads `exiting` via closure) is not in the dependency array. Pressing Escape
during the 220ms exit animation triggers a stale handler where `exiting = false`, potentially
creating a second simultaneous timeout. Guards in `unlock()` prevent visible breakage but it's a
latent bug and an ESLint `react-hooks/exhaustive-deps` violation.

**Fix**: Add `handleExit` to the `useEffect` dependency array.

**Executioner verdict**: FEASIBLE. Phase 1, same commit as P2-C.

#### ADD-3 · ToolbarV2 handleLoad passes unvalidated unknown fields to store
**Status**: CONFIRMED (Omission-2 from Verifier)  
**File**: [`src/ui/v2/layout/ToolbarV2.tsx`](../src/ui/v2/layout/ToolbarV2.tsx) ~L195

**Evidence**: `s.setColorScheme(app.colorScheme)` where `app.colorScheme` is typed as `unknown` —
no runtime validation. A malformed JSON file could corrupt store state (invalid color scheme ID
stored in localStorage, causing error on next load).

**Fix**: Add a type guard before calling store setters on loaded JSON data. Minimum:
```ts
if (typeof app.colorScheme === 'string') s.setColorScheme(app.colorScheme);
```

**Executioner verdict**: FEASIBLE. Phase 2, same ToolbarV2 commit.
