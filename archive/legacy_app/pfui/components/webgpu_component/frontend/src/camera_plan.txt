WebGPU Camera Control System Design
Requirements and Goals
•	Unconstrained Rotation: The camera should allow full infinite 360° yaw and ±90° pitch without gimbal lock or abrupt flips. Users can even turn the camera upside-down smoothly, with no sudden orientation jumps. Using quaternion-based rotations ensures continuous movement through the poles without singularities.
•	Turntable Orbit Mode: A classic CAD-style orbit around a target point (pivot). Yaw rotates around the world “up” axis (e.g. global Y axis), and pitch rotates around the camera’s local right axis. The horizon stays level (no roll). Dragging horizontally yaws around world up; dragging vertically pitches the camera up/down. This mode keeps the camera’s up vector aligned to world up except when fully inverted.
•	Arcball (Trackball) Orbit Mode: A free-form tumble around the target. The user can rotate the view about any axis by “dragging the scene” as if spinning a trackball under the cursor. This allows roll as well (unlike turntable) for maximum flexibility. The implementation must feel intuitive – mouse movements translate directly into scene rotations. (Dragging left rotates the scene left, etc., even at odd angles.)
•	Free-Look (First-Person) Mode: An optional mode where the camera isn’t locked to orbit a pivot. The camera moves and rotates freely in space (6 DoF). Typically yaw and pitch control orientation (like an FPS camera), and the user can translate the camera with keyboard (e.g. WASD for forward/strafe) or mouse controls. In free-look, there is no pivot or fixed point of rotation – the user navigates as if walking/flying through the scene.
•	Smooth Pan & Zoom: Panning (translating the view in the image plane) and zooming should be fluid and precise. In orbit modes, panning moves the camera and pivot together parallel to the view plane, so the camera continues to look at the same point from a shifted angle. Zooming is implemented as dollying the camera in and out. Ideally, implement zoom-to-cursor: when the user scrolls or pinches to zoom, the point under the cursor stays under the cursor (instead of always zooming toward the center). This reduces the need for corrective panning when zooming into details.
•	Intuitive Controls: The controls should always correspond to the user’s drag direction, even if the camera is upside-down. We must avoid the common issue where controls feel inverted after the camera passes over the top. The solution is to adjust input interpretation based on camera orientation (using quaternions and perhaps flipping the horizontal drag when inverted) so that moving the mouse left/right/up/down consistently rotates or moves the view in the expected direction.
•	Focus on Point (Re-center): The user can double-click a point in the scene to focus the camera on that point. In orbit mode, this will smoothly move the pivot to that point (and adjust camera distance if needed) so that subsequent rotations orbit around the new focus. This feature helps navigate large models by quickly zooming into an area of interest. It should animate smoothly (not jump cut) for a polished feel.
•	Robust State Management: Cleanly separate the committed camera state (the actual position, orientation, etc. that defines the view) from any transient interaction state (intermediate values during an ongoing mouse drag or animation). For example, while dragging to rotate, we might keep an initial orientation and a live delta rotation, applying the combination for display without immediately overwriting the base state until the drag ends. This prevents numerical drift from incremental updates and makes the math easier to manage. We will use a consistent representation (like a quaternion or basis vectors) for camera orientation in all modes.

Overview and Architecture
We will implement a dedicated CameraController class (in a new module, e.g. camera_controller.ts) that encapsulates all camera control logic. This controller manages the camera’s state (position, orientation, pivot, etc.), handles user input events (mouse, touch, scroll), and orchestrates smooth movements (inertia, focus animations). By centralizing camera logic, we avoid scattering state across the app, making it easier to maintain and extend.
Key architectural points:
•	Unified Camera Orientation: We represent the camera’s orientation with a quaternion (rather than Euler angles) to allow smooth, gimbal-lock-free rotation. The quaternion can be converted each frame to the direction vectors or view matrix needed by WebGPU. All rotations from user input (yaw, pitch, roll) update this quaternion. Using quaternions ensures that as the camera pitches up through 90° and beyond, it naturally rolls 180° to remain continuous rather than experiencing a sudden flip. The camera can freely spin 360° in any direction without singularities.
•	Camera Position and Pivot: In orbit modes (turntable, arcball), the camera’s position is defined relative to a pivot point (the point it’s orbiting around). We store pivot (a Vec3 in world coords) and a scalar distance (camera-pivot distance). The camera’s position is then position = pivot - forward * distance (assuming forward is the viewing direction vector derived from the orientation). In free-look mode, we don’t use a pivot or distance; the camera’s position moves independently. The controller will maintain the pivot and distance only when applicable. Switching modes may involve dropping or reinstating the pivot.
•	Camera Modes: We support three modes – "turntable", "arcball", and "free" – controlled by an enum or similar. The CameraController will interpret the same input events differently depending on the current mode:
•	In turntable mode, input drags affect yaw and pitch around the appropriate axes (global up and camera right).
•	In arcball mode, input drags result in a free-axis quaternion rotation (trackball).
•	In free-look mode, input controls behave like first-person (orientation changes yaw/pitch and we translate the camera instead of pivoting around a target).
We will provide a method to switch modes (e.g. setMode(CameraMode)), ensuring a seamless transition. When switching from an orbit mode to free-look, the camera’s current world position and orientation are retained, and we simply discard the pivot. When switching from free-look back to orbit, we may need to define a new pivot (for example, the last known pivot or a point straight ahead of the camera at some distance) – but commonly the application might default to a sensible point (like the model center or last focus point).
•	Separation of Math and State: Low-level math operations (quaternion math, vector transforms) will live in a utility module (e.g. camera_basis.ts). This keeps the controller focused on state management and input logic. For instance, functions to create quaternions from axis-angle, multiply quaternions, rotate vectors by quaternions, etc., will be defined in camera_basis.ts. The CameraController will use these utilities to apply rotations and compute new orientations or positions. Keeping math in one place makes it easier to test and ensures consistency (the same quaternion routines are used everywhere).
•	Rendering Integration: The camera controller will output the camera parameters needed by the renderer each frame. Specifically, after each update, we derive the camera’s basis vectors from the quaternion:
•	cameraForward = orientation * (0, 0, -1) (assuming forward is -Z in camera local coords)
•	cameraRight = orientation * (1, 0, 0)
•	cameraUp = orientation * (0, 1, 0)
These vectors (or the view matrix constructed from position & orientation) will be fed into the WebGPU shaders or uniform buffers. In the existing system, webgpu_core.ts uses displayCamForward, displayCamUp, displayCamRight for rendering – our controller will update these each frame to reflect the current view. We’ll remove older Euler-angle or incremental orientation code in favor of this unified quaternion approach.

Camera Rotation Modes
Turntable Orbit Mode
Turntable mode restricts camera rotation to yaw (around the world’s up axis) and pitch (around the camera’s horizontal axis), with the camera orbiting a fixed pivot. This gives a traditional orbital view where “up” in the scene remains up in the view (until you go upside-down).
•	Rotation Axes: Yaw rotates the camera around the global up axis (e.g. the Y axis in world space). Pitch rotates the camera around its own right axis (which is perpendicular to both the forward direction and world up). By combining these, the camera can move to any orientation on a sphere around the target, except that roll is locked (the camera’s up vector always stays in the same vertical plane as world up). We do not artificially clamp the pitch, allowing the camera to reach the poles (looking straight down or up at the pivot).
•	Quaternion Application: To apply a yaw and pitch update to the camera’s quaternion:
•	Compute the yaw rotation quaternion q_yaw = Quaternion(axis = worldUp, angle = Δyaw). This rotates the orientation around a fixed global axis.
•	Compute the pitch rotation quaternion q_pitch = Quaternion(axis = cameraRight, angle = Δpitch). The cameraRight axis is the camera’s local X-axis before the pitch is applied. (This axis is derived from the current orientation or from the intermediate yaw-adjusted orientation.)
•	Update orientation: newOrientation = q_pitch * (q_yaw * currentOrientation). Here we pre-multiply by yaw (global rotation) and post-multiply by pitch (local rotation). In practice, we may apply them sequentially: first apply yaw (orientation = q_yaw * orientation), then pitch about the new right axis (orientation = orientation * q_pitch). Using quaternion multiplication ensures smooth combination of these rotations. We normalize the resulting quaternion to avoid drift.
Fully using quaternions prevents gimbal lock. Even when pitch approaches ±90°, the representation remains stable. If the camera passes beyond 90° pitch (going upside-down), the quaternion will smoothly incorporate a 180° roll. This means the camera’s up vector flips without any sudden jump – it’s just part of the continuous rotation.
•	Drag Control Mapping: Dragging the mouse or touch controls the yaw and pitch:
•	Horizontal drag (Δx) → rotate yaw by some scaled angle. The scene appears to rotate left/right as the user drags.
•	Vertical drag (Δy) → rotate pitch up or down. Dragging up typically pitches the camera down (showing more of the ground), which can feel inverted, so we will adjust the sign as needed to ensure it feels natural. We likely set it such that dragging upward moves the viewpoint upward (i.e., tilts the camera to look down).
•	We will fine-tune the sensitivity (degrees per pixel dragged) to make orbiting feel responsive yet precise.
•	Pole Behavior: At the top or bottom (pitch ~ ±90°), yaw has a diminished effect on the view (rotating around world up when looking straight down doesn’t change the scene). In our implementation, as the camera passes the pole, it will start to roll around to maintain a continuous feeling. From the user’s perspective, if they drag horizontally while upside-down, we will consider flipping the direction of yaw input to avoid confusion (because when inverted, moving the mouse left might otherwise rotate the view the opposite of normal). A simple check: if the camera’s up vector dot product with world up is negative (meaning camera is upside-down), we invert the horizontal drag direction. This ensures that the controls remain consistent from the user’s viewpoint.
•	World-Up Lock: Except for the implicit 180° roll on inversion, the camera in turntable mode doesn’t roll freely. The “horizon” remains level in typical use. This is useful for technical viewing where an upright orientation is preferred.
Progress 2025-11-26: Turntable yaw/pitch inputs are now wired through the shared quaternion state (camQuat). The camera smoothly transitions through the zenith without sudden flips. Remaining tasks include tuning the drag sensitivity curves and perfecting the inverted drag logic to keep controls intuitive when the camera goes upside-down.
Arcball Orbit Mode
Arcball mode offers unconstrained rotation about all axes, using a virtual trackball metaphor. This is implemented via quaternion rotations derived from cursor motion, giving a very fluid tumble.
•	Rotation Mechanism: We use Shoemake’s classic arcball technique. Imagine a sphere centered in the view (covering the screen). When the user clicks and drags:
•	On pointer down, project the cursor’s 2D position onto this virtual sphere (or sphere cap) to get a 3D point p1. This is done by mapping screen coordinates to a [-1,1] range (normalized device coordinates), then interpreting those as x,y on the sphere and computing z = √(1 - x² - y²) (if the point is outside the sphere, we normalize it onto the sphere’s surface).
•	On pointer move, project the new cursor position to get p2 on the sphere.
•	Compute the rotation that takes p1 to p2. The rotation axis a is the cross product p1 × p2 (which is perpendicular to the plane containing both points), and the rotation angle θ can be derived from the dot product: cos(θ) = p1 · p2, so θ = arccos(p1·p2). We create a quaternion q_drag representing this rotation (axis = a (normalized), angle = θ).
•	Apply this quaternion to the camera’s starting orientation from when the drag began. If q_start is the camera’s orientation (quaternion) saved at the moment of mouse-down, then the current orientation during the drag is q_now = q_drag * q_start. We do not incrementally apply small deltas frame-by-frame; instead we always recompute the relative rotation from the initial grab point. This avoids accumulation errors and provides a stable, reversible rotation as the user moves back and forth.
•	Quaternion Composition: The key formula is q_now = q_drag * q_start. Here, q_drag is computed fresh each update from the cursor positions, and q_start is the quaternion orientation when the drag started. By multiplying, we apply the new rotation in addition to the original orientation. We will normalize q_drag and q_start as needed to ensure numerical stability. When the drag ends, we commit the new orientation by updating the camera’s stored orientation quaternion (and discarding the now-outdated q_start and q_drag). This approach cleanly separates transient rotation from base orientation.
•	Free Rolling: Unlike turntable, arcball imposes no constraint on the camera’s up vector. The camera can roll in any direction depending on how the user drags (for instance, a diagonal drag will produce a combined yaw+pitch+roll rotation). This gives maximum freedom in examining the model from any angle. The downside is that the view can become tilted, which some users might find disorienting – hence the option to use turntable mode as a more restricted alternative. We allow toggling between arcball and turntable, reusing the same camera state (the orientation quaternion will simply carry whatever orientation it had – including any roll – into the new mode, though turntable mode might ignore roll component except for the upside-down logic).
•	User Input Mapping: In arcball mode, typically the primary mouse button (left-click drag) is used for rotation (since it’s the main interaction). The movement is interpreted via the arcball math described. The feeling is that the user’s drag directly “drags the world”. We ensure the implementation is drag direction consistent: moving the mouse left makes the object rotate as if you pushed its right side backward (object appears to rotate left). The math with the cross product inherently handles this correctly.
•	Inertia in Arcball: (Detailed in a later section, but worth noting) – If the user “flings” the view (a quick drag-release), we can continue rotating with inertia. For arcball, we can derive an angular velocity vector from the last few positions (p1 to p2 changes) to continue the motion.
Progress 2025-11-26: Arcball rotation is now implemented using the stored start quaternion and live q_drag as described. The orientation updates as q_now = q_drag * q_start, which eliminated incremental drift and gimbal issues. Next, inertial rotation on release will be added so that a quick tumble continues to spin the model momentarily.
Free-Look (First-Person) Mode
Free-look mode disconnects the camera from any orbit pivot, allowing free movement and looking around from a first-person perspective. This is akin to an FPS camera or a spectator mode in 3D apps.
•	Orientation Control: We adopt a yaw/pitch scheme similar to turntable, but crucially, yaw is applied around the camera’s local up axis, not a fixed world up, since the camera might not be aligned with world axes. Typically, we will keep the camera’s “up” oriented to world up in free-look to avoid unnatural roll (most first-person cameras keep the horizon level for the viewer). We can enforce that by not allowing roll input in this mode. Yaw rotates the camera around its current up vector; pitch rotates around its right vector. This is essentially like turntable but without a pivot constraint. We will likely clamp pitch to avoid flipping over in free-look (e.g. limit to ~±85°) so the user doesn’t end up upside-down in this mode, since that’s generally not desired in first-person navigation. (If full 180° pitch is allowed, we must handle inverted controls as with turntable.)
•	Position Control: Since there’s no pivot, moving the camera is done via translations. We have two main ways to move:
•	Keyboard: In a typical setting, W/A/S/D keys move forward/left/back/right, and perhaps Q/E or R/F move vertically up/down. However, in a Streamlit environment we may not have continuous key events (Streamlit doesn’t support persistent real-time keyboard input easily). We can still plan for it (maybe via JavaScript in the front-end if needed), but we also design pointer alternatives.
•	Pointer (Mouse/Touch): We can repurpose mouse drags with a modifier for translation. For example, right-click drag (or Shift + left-drag) can translate the camera: horizontal drag = strafe right/left, vertical drag = move up/down. The scroll wheel (or touchpad two-finger scroll) can move the camera forward/backward (dolly in/out) in free-look mode. This way, all movement can be done with mouse inputs alone:
o	Scroll up = move forward, Scroll down = move backward.
o	Right-drag or middle-drag = pan the camera position (strafe or elevator).
o	Left-drag = look around (change orientation yaw/pitch).
•	No Pivot, Just Position: The camera’s state in free-look simply consists of a position vector and an orientation quaternion. We ignore any pivot or orbit distance. If the user switches from an orbit mode to free-look, we take the current camera world position (which was pivot - forward*distance in orbit mode) as the starting free-look position, and maintain the same orientation. We also internally drop the concept of pivot (or set it to null). If switching back to orbit, we might restore a saved pivot or pick a new pivot (perhaps cast a ray from camera to find an object to orbit, or use the last known pivot).
•	Movement Speed: The translation speed (per key press or per pixel drag) will be tuned. We might allow acceleration for continuous input (not applicable if keys can’t be held easily in Streamlit, but if we detect a way, e.g., hold W to move continuously forward). Since that’s tricky in Streamlit, the design relies on discrete or short interactions – e.g., each scroll tick moves a fixed amount, each drag moves proportionally to distance (we can scale movement speed by some factor relative to scene scale or use the camera’s current distance to some reference).
•	Roll (Optional): Normally free-look cameras do not roll (to keep the view upright). We will by default not include roll controls. If needed, one could implement Q/E to roll the camera, but this is low priority. The quaternion approach supports it, but we’d only use it if a use-case arises (like simulating an aircraft bank).

The code now preserves the freePosition (camera world coords) when switching modes, so toggling free-look keeps the last viewpoint. Next steps are to implement translation controls (WASD or drag) and possibly clamp pitch to avoid flipping over.
Panning and Zooming
Panning (Orbit Modes)
In turntable or arcball mode, panning moves the camera’s viewpoint laterally or vertically relative to the scene, without changing the viewing direction. This is equivalent to moving the orbit pivot on a plane parallel to the camera’s image plane, or equivalently moving both camera and pivot together in world space.
•	Mechanism: When the user drags with a pan action (commonly assigned to right mouse button drag, middle mouse drag, or left drag with a modifier like Shift), we translate the camera’s position and pivot by the same offset. The offset is computed in world coordinates from the screen movement. A typical approach:
•	Determine a scale factor so that a given screen pixel movement corresponds to a certain world distance. One method: use the current camera distance or depth to scale movements. For example, a small movement when zoomed in close should move the camera less in world units than the same screen movement when zoomed out far. We can cast a ray from the camera through the screen and find a reference depth (like the pivot distance or depth of a point under cursor) to decide how to map pixels to world units.
•	Compute the horizontal movement vector as -Δx * rightVector * k and vertical movement as Δy * upVector * k (where rightVector and upVector are the camera’s current right and up directions, and k is the scale factor). Note the negative on Δx: dragging the view to the right (positive Δx) should move the camera left relative to the scene, which is achieved by moving the pivot right (so we subtract from camera’s position movement).
•	Add this vector to both camera.position and camera.pivot. As a result, the camera and target shift together, keeping the relative orientation and distance the same, but the region of the scene in the center of view changes.
•	Result: The user perceives this as “grabbing the scene and moving it”. Importantly, because the camera’s orientation does not change when panning, the operation feels like sliding the camera’s viewpoint along a plane. We will ensure the motion is smooth by possibly applying a slight damping to panning speed to avoid it feeling too twitchy.
•	Edge Cases: If the scene is completely empty (no reference point), panning still just moves the camera in its plane. We might set an arbitrary scaling (like at some default depth). If there is a ground plane or reference, using the intersection with a plane at pivot depth under the cursor yields a natural pan scale.
•	Free-Look Panning: In free-look mode, panning is essentially just moving the camera. We can treat the same input (e.g. right-drag) as a translation command: horizontal drag moves the camera right/left, vertical drag moves it up/down. The difference from orbit is there’s no pivot to move, only the camera position changes. We use the camera’s orientation to define directions: e.g. use the camera’s right vector for horizontal movement, and either the world up or camera’s up for vertical movement. If we want to allow “flying” up/down irrespective of where the camera is looking, we might choose world up for vertical movement (so you always go straight up/down in world coordinates). Alternatively, using camera’s up will move in the direction the camera considers up (which if pitched could send you at an angle). A common FPS scheme is to keep vertical movement separate (e.g., jump/crouch keys), but for our purposes, we might stick with world up for clarity on a drag pan.
Zooming (Dolly In/Out)
Zooming changes the camera’s distance to the scene. In a perspective camera, we achieve a zoom by moving the camera forward or backward (dolly) rather than altering the field-of-view (to maintain a consistent perspective and avoid distortion issues). In orbit mode, this means changing the camera-pivot distance. In free-look, it means moving the camera along its forward axis.
•	Basic Dolly: If the user scrolls the mouse wheel or pinch-zooms on a touchpad, we compute a delta distance. For example, each notch of the wheel could reduce the distance by a certain percentage (zoom in) or increase it (zoom out). We update camera.distance = camera.distance * factor or camera.distance + delta (with some sensitivity curve). For free-look (no pivot), we move the camera position forward or back along camera.forward by some amount.
•	Zoom Limits: We enforce limits to avoid invalid positions:
•	Min Distance: We don’t allow the camera to get so close that it passes through the pivot or goes inside geometry (if the user scrolls too far in). We can set a minimum distance (maybe a small value like 0.1 units or relative to model scale). If reached, we might prevent further zoom-in or switch to a different behavior (some systems automatically transition to first-person at very close distances, but we probably just clamp).
•	Max Distance: We might set a reasonable far distance (or not, if the scene is unbounded). Possibly clamp to when the entire scene fits in view or some multiple of scene size for practicality.
The controller will clamp distance within [min, max] range.
•	Zooming Implementation: The straightforward way is:
•	In orbit: distance += zoomDelta (with sign such that positive scroll moves in or out appropriately). Or multiplicative for a smooth zoom (like factor 0.95 per tick for zoom in, 1/0.95 for out).
•	In free-look: position += forward * zoomDelta. We will choose the convention such that scrolling up (forward on a mouse wheel) zooms in (decreasing distance, moving camera closer), which is typical.
•	Zoom-to-Cursor (Screen-to-World Anchor): To make zoom more intuitive, we will implement zoom-to-cursor behavior. Instead of always zooming toward the screen center, we zoom toward the point under the cursor:
•	Determine Anchor Point: On a zoom event, cast a ray from the camera through the cursor’s position on the screen into the 3D scene. If we have access to the scene’s geometry or a depth buffer, we try to find the intersection point of this ray with an object or terrain. That intersection is the anchor point – the thing the user is attempting to zoom in on. If an intersection is found, we use it. If not (e.g., the user is pointing at empty sky or we lack picking), we fall back to a reasonable choice: for instance, use a point on the camera’s forward ray at the current pivot distance (meaning assume they intend to zoom towards where the camera is looking at that depth).
•	Compute Camera Movement: We want to move the camera closer or farther such that the anchor point stays under the cursor during the zoom. If we only dolly forward, the anchor point (if off-center) will drift toward the center. To counteract this, we combine dolly with a slight pan. One method:
o	Before zoom, note the screen position of the anchor (which is at the cursor by definition initially).
o	Determine the new desired distance: e.g. newDistance = currentDistance * factor (for orbit) or moveAmount for free-look.
o	If in orbit mode, set camera.distance = newDistance. This moves the camera along forward vector.
o	Now, to keep the anchor in the same screen spot, we adjust the pivot: essentially, we want the anchor to be along the camera’s center line after the move. We can set the new pivot equal to the anchor point (so that the camera will now orbit around that point). However, switching the pivot on every tiny zoom step might be too aggressive and cause a slight lateral shift. Another approach is to interpolate: move the pivot a fraction of the way toward the anchor. But the simplest reliable way: recenter on the anchor for the zoom action. So we set camera.pivot = anchor and then position = pivot - forward * newDistance. This way, after zooming, the anchor point is exactly at the center of view (under the crosshair).
o	If we don’t want to fully center it (to truly keep it under the cursor, not move it to center), a more complex calculation would be needed: essentially, figure out how the camera’s view frustum changes with the new distance and shift the pivot accordingly such that the projection of the anchor remains at the same normalized coordinates. This can be done via some algebra using the projection matrix. But in practice, snapping the pivot to the anchor might be an acceptable solution for now, as it effectively makes the anchor the new center of rotation going forward.
•	Fallback: If no anchor could be determined (ray hit nothing and perhaps we don’t have a known scene depth), we can default to zooming toward the center of the view (keeping pivot same). Or we can use the depth of the current pivot as an approximate anchor plane.
•	Effect: With zoom-to-cursor, if the user points at a specific feature on the model and scrolls in, that feature stays under their cursor and zooms closer, rather than sliding off toward the center. This feels much more natural. CAD applications and Google Earth use this technique. We will likely enable this by default. It does mean the pivot (target) may shift during zoom, but the user may not notice except that the zoom felt intuitive.
•	Orthographic Note: (If we ever use orthographic projections, zooming would adjust the camera’s orthographic scale rather than moving it. The concept of zoom-to-cursor still applies: you would adjust the projection center or use an offset to keep the cursor’s world point stationary in the view.)

Inertia and Smooth Motion
To give the camera controls a professional, smooth feel, we incorporate inertia. Inertia means that when the user releases the mouse after a quick drag, the camera continues moving slightly, as if it has momentum, and then gradually slows to a stop. This applies to rotations (orbit or free-look), panning, and zooming if applicable.
•	Rotation Inertia: During a rotation drag (turntable or arcball), we track the angular velocity. For turntable, we might track separate velocities for yaw and pitch (in degrees per second). For arcball, we can derive an axis-angle velocity. For example, as the user moves the mouse, we can compare the last few orientation changes or cursor positions over time to compute an estimated angular velocity vector (axis + speed). When the drag ends (pointerup), if the angular speed is above a threshold, we enable inertial rotation: the camera will keep rotating at that speed, decelerating over time. For turntable, this means continue changing yaw/pitch each frame by a decaying amount. For arcball, we could continuously rotate the camera around the derived axis by the stored angular velocity (using quaternion slerp or small incremental rotations each frame). We apply a damping factor each frame (e.g. multiply velocity by 0.95 or use an exponential decay function) so that the motion eases out and stops smoothly.
•	Pan Inertia: Similarly, if the user was panning and releases while still moving the mouse quickly, we carry on translating the view. We store a 2D velocity for the pan (screen-space or world-space units per second). On release, we convert that to a world movement vector (in the camera’s plane) and move the camera & pivot by that vector each frame, reducing it gradually to zero. This allows the view to coast to a stop instead of halting abruptly.
•	Zoom Inertia: For scroll wheel zoom, inertia usually isn’t needed because scroll is typically discrete steps. However, for touchpad pinch-zoom or a kinetic scroll, one might implement it. If we detect a “fling” zoom (e.g., in a touch environment where the user does a quick pinch release), we could continue adjusting the distance for a brief moment. This is less common, but we’ll keep the architecture open to it.
•	Implementation: The CameraController will have an update loop (likely tied into the app’s animation frame or update cycle) where it checks for any ongoing inertial motion and updates the camera state accordingly:
•	We maintain state like inertiaRotationVel (could be a vector or separate yaw/pitch velocities), inertiaPanVel (a 2D or 3D vector), inertiaZoomVel (a scalar for dolly speed).
•	On each frame, if any of these are non-zero, we integrate them: e.g. camera.orientation = rotationVelocityAxisAngle * camera.orientation (for rotation), camera.pivot += panVelocity, camera.position += panVelocity (for orbit pan), or camera.position += forward * zoomVel (for free-look zoom).
•	After applying, multiply each velocity by a damping factor (or subtract a fixed deceleration). If the magnitude falls below a small threshold, we set it to zero (stop the motion).
•	If the user interacts again (e.g., another drag or click) during an inertia motion, we cancel the inertia immediately to give priority to the new interaction. For example, calling markInteraction() could clear any ongoing inertial velocities.
•	Duration and Feel: The exact damping will be tweaked. A common approach is an exponential decay that slows movement over perhaps 0.5–1 second. The idea is to mimic a friction that is just enough to stop the motion in a short time, without feeling abrupt. We also ensure frame-rate independence by scaling motion by the actual deltaTime between frames (for consistent behavior across 60Hz vs 144Hz, etc.).

Focus and Pivot Re-Centering (Double-Click Focus)
To enhance navigation, the user can double-click on a point in the scene (or tap twice) to refocus the camera on that point. This is often called “focus on object” or “zoom to fit selection” in other applications. Our implementation smoothly moves the camera so that the clicked point becomes the new pivot (center of rotation).
•	Raycast to Find Point: On a double-click event, we cast a ray from the camera through the 2D canvas at the click position into the scene. We need either the actual geometry or a depth buffer to get the intersection. Assuming we can get it (in a Streamlit context, perhaps the 3D engine can expose a picking function), we retrieve the 3D coordinates of the clicked point on the model. If nothing is hit (e.g., background was clicked), we might choose to do nothing, or we could default to focusing the camera on some fixed point (like the origin or current pivot).
•	Animating the Transition: Instead of teleporting the camera instantly, we perform a short animation (say 0.25–0.5 seconds) to move the camera to the new pivot. This involves:
•	Pivot change: Set the target pivot to the clicked point.
•	Distance adjustment: Optionally adjust the camera distance for a better view. Often, if the clicked point is far from the current pivot, we want to maintain the same distance. But if the clicked point is on the surface of the object and the current distance is very large or very small relative to it, we might want to zoom in or out to frame that point nicely. A strategy: compute a desired distance such that the point is a certain fraction of the view (perhaps based on scene size or just keep the same distance unless very close to camera).
•	Position/orientation interpolation: We need to move the camera’s position from its current position to the new position (which is newPivot - forward * newDistance for orbit cameras) while changing the pivot and keeping the point in view. The orientation likely should change to look at the new pivot. If we keep the same forward direction, the camera might already be roughly facing that way, but if not, we can interpolate orientation as well to directly look at the new target.
•	We will use an easing function (like ease in/out) to make the motion smooth. This can be done via a simple linear interpolation of position and perhaps spherical interpolation (slerp) for orientation.
•	Focus Tween Implementation: The controller will provide a function like startFocusTween(targetPoint: Vec3). When called, it will:
•	Compute the current state (position, orientation, pivot, distance).
•	Compute the desired target state (new pivot = targetPoint, new distance maybe equal to current distance or adjusted, and orientation such that camera forward aims at target pivot from the current approach direction).
•	Start a timer or use the animation loop to gradually transition from current state to target state over a fixed duration (e.g. 0.3 seconds).
•	During the tween, user input might be locked out or if the user does interact, we will cancel the tween immediately (to let them take control — nothing is more jarring than fighting the camera).
•	At the end of the tween, set the camera’s committed state to the new pivot and position, ensure orientation is precisely adjusted to face the pivot.
•	Canceling/Interrupting: If a new user interaction begins while focusing (e.g., they drag the view mid-tween), we should abort the focus animation. A method cancelFocusTween() will stop the animation and keep the camera at its current state (partway through the transition). We’ll integrate this by calling CameraController.markInteraction() on any user input which in turn calls cancelFocusTween() internally to ensure the camera immediately responds to the user.
•	Free-Look Mode Behavior: In free-look (which has no pivot), double-click-to-focus could be handled differently. We might choose to interpret a double-click as a command to look at the point without moving the camera to it. For example, we could rotate the camera in place so that its forward vector points at the clicked location (essentially a quick orient toward that point). However, we won’t move the camera’s position because that would be a teleport. Alternatively, we might simply not enable this in free-look mode (or treat it as a no-op). The primary use of focus is in orbit modes.
•	Home/Predefined Views: (Not exactly part of double-click, but related) We could have preset buttons to reset the view or set an orthographic view. This can be achieved by setting specific pivots/orientations in the controller as well (since our controller can take arbitrary targets).
Progress 2025-11-26: Double-click focus is implemented. On double-click, a raycast finds the clicked point on the mesh. The controller calls startFocusTween(hitPoint) which creates a 260 ms eased animation. During this tween, the camera smoothly moves to center that point: the pivot shifts to the point, and the distance is adjusted based on the hit depth (so the point is comfortably framed, not too close or too far). The animation updates the camera position and orientation each frame. If the user intervenes (e.g., another drag), CameraController.markInteraction() triggers cancelFocusTween(), aborting the animation to give control back to the user immediately. The result is a slick “zoom to selection” feel when double-clicking, with no abrupt jump.
State Management and Interaction Lifecycle
A clear separation of camera state vs interaction state is critical. We define structured types to hold the camera’s persistent state, and use temporary variables for ongoing interactions.
•	CameraState: This represents the committed, authoritative state of the camera at any given time. We can define it as an interface or type:
type CameraMode = 'turntable' | 'arcball' | 'free';

interface CameraState {
  mode: CameraMode;
  position: Vec3;    // Camera world position
  orientation: Quat; // Camera orientation (rotation) as a quaternion
  pivot?: Vec3;      // Orbit target point (undefined or null in free mode)
  distance?: number; // Distance from camera to pivot (for orbit modes)
  fov: number;       // Field of view (if needed for projection)
  // ... potentially near/far clip distances, etc.
}
In orbit mode, pivot and distance are defined; in free mode, those may be unused (or pivot null). The orientation is always a quaternion. The position is always the camera’s world position (even in orbit mode, it’s derivable from pivot+distance, but we store it for convenience). We keep mode to know how to interpret inputs.
•	Display vs Committed Orientation: To avoid accumulating floating-point error and to manage smooth dragging, we may use two quaternions for orientation:
•	state.orientation is the base orientation (committed when no interaction is happening).
•	A separate display orientation (or a combination of dragStartOrientation and a current drag delta quaternion) is used during interactions to show intermediate results. For example, when the user is mid-drag in arcball, we don’t immediately overwrite state.orientation every tiny move. Instead, we keep dragStartOrientation = state.orientation (saved at drag begin), and compute q_now = q_drag * dragStartOrientation each frame for display. We might store displayOrientation = q_now just for convenience to pass to rendering. Only when the drag ends do we set state.orientation = q_now officially. This ensures if the user keeps dragging around and around, we’re always composing from the original orientation, not the last incremental step (which could introduce tiny errors or drift).
•	Similarly for turntable: we might store startYaw and startPitch at drag start, update temporary yaw/pitch as they drag, and only commit final angles at the end. But since we use quaternions, we can do it with quaternions entirely as well.
•	The CameraController will manage these transient vs committed values internally. From the perspective of the rest of the app (rendering, etc.), it can expose the effective orientation (which is displayOrientation if an interaction is ongoing, or the committed orientation otherwise).
(Progress 2025-11-26: The implementation now mirrors this approach. The code uses a camQuat for state and a displayCamQuat for the transient orientation during drags/tweens. This way, what’s sent to the shader each frame is always the up-to-date view (display quaternion), but the underlying camQuat only updates when an interaction is finalized or when inertia/tween ends. This prevents jitter and precision loss over long sessions of dragging.)
•	Transient Interaction State: Inside the CameraController, we will have fields to track ongoing interactions:
•	dragStartPos: the initial pointer coordinates when a drag begins.
•	dragCurrentPos: last known pointer coords (for calculating deltas or velocity).
•	dragStartOrientation: a quaternion snapshot of the camera’s orientation at drag start (for arcball/turntable rotation).
•	dragStartPivot: (Vec3) pivot at drag start, in case needed (for example, if we wanted to allow moving pivot mid-drag for some reason).
•	currentDragQuat: for arcball, the quaternion representing rotation from dragStartPos to current position (q_drag as described earlier).
•	isDragging: boolean state, possibly with an enum for what action is being dragged ('rotate' | 'pan' | 'zoom').
•	Velocity trackers: e.g. lastMoveTime, lastOrientQuat or lastAngleDelta to compute rotation speed; panVelocity, rotVelocity, etc.
•	Inertia state: as mentioned, store ongoing velocity or angular momentum.
•	Focus tween state: maybe store tweenActive boolean and tween target values if an animation is in progress.
These variables are used during an interaction but are not part of the persistent camera state. They get reset/cleared when the interaction ends.
•	Pointer Event Lifecycle: The high-level flow for pointer interactions:
•	PointerDown (mouse down / touch start): Determine what action to start based on the input (which button or touch gesture). For example:
o	Left click (with no modifiers) → begin rotate (orbit or free-look) action. Save dragStartOrientation = state.orientation (or for free-look maybe just start tracking yaw/pitch).
o	Right click or Shift+left → begin pan. Save initial position for reference.
o	Middle click (or Ctrl+left) → maybe also pan (depending on user preference).
o	Touch with two fingers → could begin pinch (zoom) or two-finger drag (pan).
o	Double-click is a separate event, not handled in down but via a click handler, which triggers focus.
o	Record dragStartPos (initial cursor position).
o	Set isDragging = true and the current action mode.
o	Call markInteraction() to note that user is actively interacting (this could, for example, pause any idle animations or analytics of inactivity).
•	PointerMove (mouse move / touch move): If isDragging:
o	Compute delta from last recorded position (or from start, depending on action).
o	If rotating:
o	In turntable: translate pixel delta to yaw/pitch angles. For example, Δx pixels -> Δyaw = Δx * rotationSpeedFactor, Δy -> Δpitch. Use these to compute a new quaternion as described in Turntable section, by applying to dragStartOrientation or accumulating changes. Update the displayed orientation (but maybe keep the base in dragStartOrientation until done).
o	In arcball: compute q_drag from projecting start and current positions to sphere. Set display orientation = q_drag * dragStartOrientation.
o	If panning:
o	Compute movement in screen and map to world offset (as in panning section). Update state.pivot and state.position continuously for smooth feedback (since panning doesn’t have the same “commit later” issue – we can apply incremental, it’s just linear).
o	If zooming (e.g., pinch zoom):
o	Compute zoom delta (from touch pinch scale or maybe vertical mouse movement if we had a drag-to-zoom mode). Adjust distance accordingly.
o	If implementing zoom-to-cursor dynamically on drag-zoom, we’d do similar anchor calculations continuously (this is complex for drag, so likely we stick to scroll events for zoom, not click-drag for zoom).
o	Update dragCurrentPos and possibly store recent history for inertia (e.g., store last few moves with timestamp to derive velocity).
o	Each move, call a function like updateDisplayCamera() to apply these changes to the view (update displayCamQuat, position, etc.). The WebGPU view will use the updated display orientation/position on the next redraw.
•	PointerUp (mouse up / touch end):
o	End the drag. Determine if we should start inertia:
o	If the action was rotate or pan and we have velocity, calculate it and store in the inertia state.
o	For example, if turntable rotate: compute degrees per second for yaw and pitch from last few moves; store in e.g. inertiaYawSpeed, inertiaPitchSpeed.
o	If arcball rotate: derive an axis and angular speed from the final q_drag vs second-to-last, etc.
o	If pan: compute velocity vector.
o	If no significant velocity (user released slowly), then no inertia; just end immediately.
o	For rotation, commit the orientation: set state.orientation = current display orientation (if not already updating state continuously). In our approach, we might have already updated it as we went, but if not, do it now. Normalize quaternion.
o	Ensure state.position is consistent (for orbit, we can recalc position = pivot - forward*distance after orientation change).
o	Clear isDragging and related drag state variables.
o	If inertia was set, a flag like isInertiaActive = true will keep the update loop running.
o	Also possibly call markInteraction() here to mark the end of user interaction (if we want to record last interaction time, etc.).
•	Keyboard Events: If we implement any (like WASD for free-look), those would feed into methods like handleKeyDown/Up which adjust the camera position/orientation accordingly. In Streamlit, we don’t have a persistent keydown event loop easily, so this may be minimal or omitted. The design ensures all needed movement can be done with pointer events (e.g., use scroll for forward/back instead of "W/S").
•	markInteraction(): This is a utility in the controller to mark that a user interaction occurred. In practice, this might:
•	Cancel any ongoing focus tween (cancelFocusTween()).
•	Cancel inertia if needed (or at least know that user is now controlling, so inertia will be overwritten).
•	Possibly record a timestamp of last interaction (could be used to hide UI after inactivity or something, but that’s outside scope).
•	In our context, it primarily ensures smooth override of automated motions by user input.
(Progress 2025-11-26: Pointer event handling has been fully moved into CameraController. The onPointerDown, onPointerMove, onPointerUp methods replace the old event logic in webgpu_core.ts. This consolidates the state tracking described above. We verified no regressions: the camera still orbits/pans/zooms as before. Now that the lifecycle is centralized, adding features like inertia or new input modes is much easier. markInteraction is called on any user-initiated event to promptly cancel tween animations or inertia – ensuring the user never fights the camera for control.)

Mode Switching and Intuitive Controls
We provide a way to toggle between camera modes (turntable, arcball, free-look). This could be via a UI control or programmatic API. The transition rules:
•	Switching to Free-Look: We drop the pivot. The controller can simply keep the camera’s current world position and orientation, set state.mode = 'free', and null out state.pivot. The state.distance is no longer used. Essentially the camera is now free-floating at whatever position it was at that moment. To avoid confusion, we might also want to adjust orientation to ensure no unintended roll carries over (since free-look typically expects a level horizon). If the orientation has a roll (from arcball), we could optionally remove it by realigning the camera’s up to world up upon entering free-look – but this could be jarring, so it might be better to keep as-is and let the user reorient if needed. We will likely keep orientation unchanged to respect what they were looking at.
•	Switching from Free-Look to Orbit: We need a pivot. One approach: cast a ray from the camera forward direction to find an intersection in front of the camera (maybe the model or ground). If found, use that point as the new pivot and set distance accordingly. If not, we could use a default pivot (for instance, a point some default distance ahead of the camera). Alternatively, we could remember the last pivot the camera had before going free-look and reuse that (although the camera might have moved far from that). A pragmatic approach: if the user switches back to orbit, we set pivot = camera.position + forward * defaultDistance (so the pivot is a point in front of the camera). This ensures the camera doesn’t end up with pivot at its position (which would break things). We set state.mode = 'turntable' or 'arcball' as desired, and reinstate that pivot. The effect is the camera will now orbit around that new pivot point.
•	Switching between Turntable and Arcball: These both are orbit modes, so pivot and distance remain the same. We simply change how drags are interpreted. If the camera orientation currently has some roll (possible if coming from arcball), and we switch to turntable, the roll will effectively get ignored in control (since turntable won’t intentionally roll the camera further, except the implicit flip handling). The camera will still be at that orientation though. It might feel odd if the camera is tilted and now only yaw/pitch moves it. Some software forcibly level out the camera when switching to turntable (i.e., realign camera up to world up). We could decide to do that: on switching to turntable, apply a corrective roll rotation to set camera.up parallel to world up. This would preserve the same view direction but “unscrew” the tilt. This might be a good default to avoid confusion. On the other hand, doing so could be a surprising jump. We could also leave it and assume if the user chose arcball they’re ok with tilt, and if they go back to turntable we might just leave the tilt until they adjust pitch past a pole which will then correct it. This detail can be refined in testing. The design will note this as a consideration.
•	Maintaining Intuitive Controls: As emphasized, when the camera is upside-down or in an odd orientation, we adjust how we interpret input to keep it intuitive:
•	We already plan to flip horizontal drag in turntable if upside-down.
•	In arcball, the relationship is inherently direct via the math, so it’s always correct relative to the sphere drag (no extra flip needed).
•	In free-look, if we limited pitch to ±90°, we never go fully upside-down. If we didn’t and allowed 180°, then we’d similarly want to handle inversion (but we’ll likely clamp to avoid it).
•	We ensure that panning directions always correspond: dragging the scene left moves it left etc. This is naturally handled by our use of the camera’s right/up vectors appropriately.
•	We will thoroughly test scenarios like flipping the camera and then panning, to ensure the controls don’t suddenly invert in feel (aside from the purposeful flip logic which prevents a wrong-feeling inversion).
•	Preventing Mode Glitches: If the user switches modes mid-drag or mid-inertia, we will handle it gracefully:
•	If switched mid-drag, we might end the drag as if the mouse was released in the old mode (to avoid applying the old mode’s logic in the new mode unexpectedly). The user can then press again to start a new interaction in the new mode.
•	If switched while inertia is ongoing, likely we cancel the inertia (different modes have different kinds of inertia – carrying it over could be weird). For example, if an arcball spin was continuing and user toggles to turntable, probably best to stop the spin. The setMode() could call cancelInertia() (which we can implement by zeroing velocities or using markInteraction() internally since mode switch is an intentional user action too).
•	Focus tween – if mode is switched during an auto-focus animation, we also cancel the tween (and maybe instantly jump to end or just stop where it is). This is an edge case but likely rare to switch during an auto movement.

**1) Architecture & State Consolidation (High Priority)**
- **Why:** State is split across multiple modules (webgpu_core.ts, webgpu_preview.ts, camera_controller.ts) and host defaults are merged into `cfg`, which causes forced resets.
- **Changes:**
  - Make `CameraController` the canonical owner of camera state (committed + display).
  - Move `applyCameraPayload`, `cameraPayloadDiffers`, `pendingForceCameraPayload`, `localControlLastAt`, and the grace/delay behavior into `CameraController`.

  - Expose `CameraController` API: `setPayload(payload, {force?: boolean, forceFull?: boolean})`, `markInteraction()`, `onPointerDown/Move/Up`, `startFocusTween(...)`, `cancelFocusTween()`.

**2) Payload Handling & Debug API (High Priority)**
- **Why:** Using `cfg` to compute `forceCamera` causes host update semantics to be wrong.
