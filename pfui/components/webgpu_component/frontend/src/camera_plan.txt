Professional-Grade WebGPU Camera Control System Design
Requirements and Goals
We need to implement a robust camera controller for a WebGPU 3D viewer that supports multiple interaction modes and fluid user experience. Key goals include:
•	Unconstrained Rotation: Full 360° yaw and 180° pitch rotation without artificial limits. The camera should be able to flip upside-down when passing the zenith/nadir (pole) without abrupt orientation jumps or gimbal lock issues[1].
    Progress 2025-11-26: Quaternion math now lives in `camera_basis.ts` and both committed/display quaternions are tracked in `webgpu_core.ts`, so pole crossings stay continuous in every mode.
•	Turntable Orbit Mode: A CAD-style turntable control where yaw rotates around the global vertical (world-up axis) and pitch rotates around the camera’s local right axis. The horizon remains level (no roll), and drag directions feel natural (dragging horizontally yaws around world up; dragging vertically pitches up/down)[2].
    Progress 2025-11-26: Turntable basis mutations now route through the shared quaternion state, keeping roll suppression intact while presets/payloads stay in sync; further UX tuning still needed.
•	Arcball (Trackball) Mode: A free-form orbit mode (toggleable via UI or props) that lets the user tumble the view around the target as if dragging a trackball. This allows rotation about all axes (including camera roll) for maximum flexibility[2]. It should be intuitive – moving the mouse rotates the scene as if you’re “rolling” a sphere under the cursor.
    Progress 2025-11-26: Pointer-down captures now stash the start quaternion and compute drag deltas as `q_drag * q_start`, eliminating incremental drift; inertia remains to be wired.
•	Free-Look Mode: An optional first-person or “fly” mode where the camera is not constrained to orbit around a fixed target. In free-look, the camera’s position and orientation move freely (no locked pivot). This mode enables inspecting the scene from any position, with controls similar to FPS games (e.g. click-and-drag to look around, WASD to move).
    Progress 2025-11-26: Free-look reuses the same quaternion/basis pipeline so switching modes preserves heading; keyboard translation and pitch clamping are next.
•	Smooth Pan & Zoom: Panning (translating the view) and zooming should be fluid and precise. Panning will translate the camera or orbit target without changing orientation, and zooming (dolly in/out) should preferably be zoom-to-cursor – meaning the point under the cursor stays centered while zooming[3]. This avoids the need for constant panning adjustments.
•	Intuitive Controls: User input mapping should be consistent across modes. No unexpected inversions of controls when the camera passes over the top – the motion should always correspond to the drag direction on screen. We must avoid the common “orbit flip” problem when looking straight up/down by using a rotation representation that remains continuous (quaternions)[1]. Dragging the mouse left/right or up/down should have a predictable effect in all modes.
•	Focus on Point (Re-center): Users can double-click or tap a point in the scene to refocus the camera on that point. In orbit modes, this means shifting the orbit target to that location (and possibly adjusting camera distance) so that subsequent rotations pivot around the new point. This feature enhances navigation by letting users quickly center on areas of interest[4].
•	State Management: Cleanly separate transient interaction state from the committed camera state. For example, during an ongoing drag, maintain a temporary rotation offset that is applied to the camera, and only commit it when the drag ends. This approach, consistent with our existing displayCamForward/right/up basis usage, prevents accumulating numerical error and makes the math easier to manage. The camera’s basis (forward, right, up vectors) should be updated only with committed changes, while a separate variable represents the in-progress change. (This mirrors known arcball implementations that keep an initial quaternion and a live quaternion during drag[5].)
    Progress 2025-11-26: Display basis/quaternion now mirror transient drags while commits normalize into `state.cam*`, so payloads/presets no longer accumulate float drift.
Overview of the Solution
We propose a CameraController module in TypeScript that encapsulates all camera interaction logic. This controller will manage the camera’s state (position, orientation, target pivot, etc.) and expose methods to handle user input (pointer events, gestures, etc.). The design cleanly separates the mathematical operations (in camera_basis.ts) from event handling and state updates (in webgpu_core.ts or a new controller class), in line with PotFoundry’s basis-based camera philosophy. The core ideas are:
•	Use a unified camera representation based on position and orientation (using a quaternion or equivalent basis) rather than Euler angles. A quaternion provides smooth, continuous rotations without gimbal lock[6], allowing the full range of motion (360° yaw, 180° pitch) with no singularities. We will convert this orientation to the existing forward/right/up basis vectors for rendering usage each frame.
•	Maintain a distinct camera pivot (target point) for orbit modes. The camera’s position in orbit mode is derived as position = pivot - (forward * distance), where forward is the camera’s viewing direction. In free-look mode, the concept of pivot is not used (the camera moves independently).
•	Implement mode switching between Turntable, Arcball, and Free-look via a mode flag or enum. The CameraController will interpret input events differently depending on the mode, but will use shared data structures so switching modes is seamless. (For example, if switching from orbit to free-look, we can drop the pivot but carry over the camera’s current orientation and position.)
We will update the existing webgpu_core.ts to replace any old orbit/tumble logic with calls into this new CameraController. The camera_basis.ts utilities might be extended to support new operations (like constructing quaternions, transforming basis vectors, etc.), but we will strive to keep the math helper functions separate from state management. Below we detail each aspect of the implementation.
Camera Rotation Modes
Turntable Orbit Mode
In turntable mode, the camera orbits around a fixed target point (pivot) like a potter’s wheel or turntable. The implementation will maintain spherical angles under the hood (or derive them from quaternions): a horizontal angle (yaw) around the global up-axis, and a vertical angle (pitch) around the camera’s right axis. We do not clamp the pitch to avoid the poles; instead we allow it to reach ±90° (straight up or down). At exactly the poles, yaw rotations technically have no effect on the view orientation, but as soon as the camera passes the pole, it will effectively be upside-down. By using quaternions for orientation, we ensure the transition is smooth – the camera will roll 180° implicitly rather than suddenly flipping the view vectors[1]. In practice, this means the camera’s up vector will invert once you go beyond 90°, but it happens as a natural consequence of the continuous rotation representation, not an abrupt forced flip.

    Progress 2025-11-26: Turntable yaw/pitch inputs now update `camQuat` and `displayCamQuat` via the shared helpers, so pole crossings stay smooth; still need to finalize drag-speed curves and flip detection UX.
Rotation controls: Dragging the pointer will adjust yaw and pitch: if the user drags to the right, we rotate the camera to the right around the world’s vertical axis; dragging up rotates the camera up around the camera’s side axis. The drag-to-rotation mapping will be set so that it feels direct. (For example, dragging upward causes the scene to tilt upward, showing more of the ground – which means the camera pitched downward. We’ll ensure this is intuitive by possibly inverting the pitch input if needed.) Because turntable locks the roll, the camera’s “up” remains roughly aligned with world up except when fully inverted. This mode is essentially the classic orbit control with a defined up-direction[2], giving users a consistent frame of reference (no confusion about which way is up).
Internally, each drag will create a small yaw rotation (about global Y axis) and/or pitch rotation (about camera’s right-axis). We compose these via quaternion multiplication with the current orientation. For example: newOrientation = rotate(quat, axis=worldUp, angle=Δx) * rotate(quat, axis=cameraRight, angle=Δy) * currentOrientation. Using quaternions avoids gimbal lock and axis discontinuities by smoothly interpolating the orientation in 3D space[6]. We also ensure the turntable’s implementation handles the pole crossing elegantly: as the pitch approaches ±90°, further mouse movement will start to introduce a camera roll so that yaw dragging still causes a rotation in the viewer’s plane rather than doing nothing or snapping axes. This way, even when the camera goes upside-down, continuing a left-right drag will appear to rotate the scene consistently (albeit the camera is inverted). The end result is a full 180° pitch range without any locked viewpoint, and no sudden camera flips – any inversion happens as a smooth continuation of the drag.
Arcball (Trackball) Orbit Mode
Arcball mode provides unconstrained rotation about all axes by treating the scene like a trackball[2]. When the user clicks and drags, we map the mouse motion to a virtual sphere (or hemisphere) centered on the viewport. Each drag motion from the initial press point to the current point corresponds to a 3D rotation. We will implement Shoemake’s arcball algorithm: on mouse-down, record the initial click’s projection onto the virtual sphere (point p1). On each mouse move, project the new cursor position to the sphere (point p2). Compute the rotation that takes p1 to p2: the axis of rotation is a = p1 × p2 (cross product) and the angle θ between p1 and p2 is given by the dot product (cos θ = p1·p2)[7][8]. We then create a quaternion representing that rotation and apply it to the camera’s orientation. Importantly, we compose this rotation with the camera’s starting orientation at drag-begin (which we saved). For example, if q_start is the quaternion of the camera when the drag began, and q_drag is the quaternion from p1 to p2, then the new camera orientation q_now = q_drag * q_start. This ensures the rotation is applied relative to the camera’s original orientation at the start of the drag (so small drags always produce small relative rotations). We do not accumulate incremental Euler angles; we always compute the delta from the initial drag point to current, which is more stable. This approach cleanly separates the transient rotation (q_drag) from the base orientation (q_start), matching recommended arcball practices (e.g. storing an mQDown and updating mQNow on the fly[5]). When the drag ends, we multiply the base orientation by the final q_drag to commit the new orientation as the starting point for future interactions.

    Progress 2025-11-26: Implemented the q_drag * q_start workflow with normalized display bases; next actions include refactoring into a controller class and adding inertial continuation.
Arcball mode allows the camera to roll freely – so unlike turntable, the camera’s up vector is not constrained to world up. The benefit is that the user can view the scene from any angle without gimbal issues or gimpy poles. However, it could be disorienting to some users if the view tilts. We will provide a UI toggle or programmatic switch to choose Arcball vs Turntable, so users (or the app) can select their preferred mode. The code structure might treat both as orbit modes but with a conditional for how rotations are computed: turntable uses separate yaw/pitch constraints, arcball uses the trackball quaternion approach. Toggling modes can reuse the same camera state (position, orientation, pivot) – for example, enabling arcball simply means subsequent drags use the trackball algorithm, but the current orientation carries over (even if it includes a rolled state).
Free-Look Mode (First-Person)
Free-look mode divorces the camera from any orbit target. Here, the camera has full six-degree-of-freedom movement: it can yaw/pitch (and potentially roll) in place, and it can translate in any direction. This is akin to a spectator camera or FPS camera. We will implement free-look rotation similarly to turntable’s approach (yaw and pitch angles), but crucially, the yaw will be about the camera’s own up axis (not the world’s up) since in free-look the camera’s up can drift if roll is allowed. We likely will not intentionally introduce roll in free-look unless a specific UI for rolling is added – typically, free-look (FPS style) locks roll to keep the horizon level for the user, so we could treat the camera’s local up as aligned to world up initially and only allow yaw (360°) and pitch (~180°) like an FPS camera. This gives a stable experience (no sideways tilt unless the user specifically wants to simulate an aircraft roll, which is an edge case).
For translation, we’ll support keyboard or on-screen controls: e.g. W/A/S/D to move forward/left/back/right in the camera’s local coordinate system, and perhaps Q/E or R/F to move up/down. In a Streamlit context, keyboard events may be tricky, but we can also use right-drag or a modifier + drag for movement. Another approach is to use the scroll wheel in free-look to move forward/back (instead of dollying toward a pivot, it literally moves the camera forward). This will be configured to ensure a fluid free-fly experience.

    Progress 2025-11-26: Free-look rotations now reuse the orbit quaternion utilities, and payload/preset switches preserve `freePosition`; translation controls are still TODO.
Implementationally, enabling free-look mode means we ignore any target/pivot. The camera’s position and orientation are updated directly. Rotations are applied to the camera’s orientation quaternion (similar to turntable, but with no external pivot). Panning in free-look is effectively the same as translating the camera: for example, dragging the scene (with a certain mouse button or modifier) could strafe the camera horizontally/vertically relative to the view. Zoom inputs (scroll) in free-look will translate the camera forward or backward along its forward vector (achieving a similar effect to moving closer or farther).
Because free-look can potentially lead to the camera being upside-down as well (if the user pitches past 90°), we will again rely on quaternions to handle the continuity. If we decide to prevent roll in free-look (to mimic typical FPS controls), we will clamp the roll or realign the camera’s up after rotations. Otherwise, if we allow full 360° pitch (which would loop the camera), the controls might invert after passing 180° of pitch – but since free-look is less frequently used for continuous upside-down loops, this is a minor consideration. By default, we can limit pitch to ±90° in free-look to avoid going upside down, which is a reasonable UX choice for an FPS-style camera. In any case, free-look uses the same underlying math utilities and state structure, just with pivot undefined and position/orientation updating freely.
Panning and Zooming Mechanics
Panning (Orbit Modes): In turntable or arcball mode, panning will strafe the camera and pivot together in the view plane. Typically, orbit controllers pan by shifting the target (pivot) horizontally/vertically relative to the camera. We will implement panning such that a click-and-drag with e.g. the right mouse button (or a two-finger drag on touch, or left-drag with a key modifier) moves the camera’s target along the plane parallel to the camera’s image plane. This means if the user drags left, the entire view’s center shifts left – effectively the camera and pivot move right relative to the scene. Concretely: we can take the camera’s right and up direction vectors, and move the camera’s position and pivot by a vector Δx * -right + Δy * up (with appropriate scaling for speed and possibly adjusted for depth). This translates both the camera and pivot so that the camera continues looking at the same point but that point shifts in screen space. Because the camera orientation doesn’t change during a pure pan, the operation is straightforward addition to position and target. We will include speed damping so that panning feels smooth and not overly sensitive.
Panning (Free-Look): In free-look, panning is essentially the same as moving the camera. A horizontal pan drag could move the camera laterally (strafing), and a vertical pan drag could move it up/down. We’ll use the camera’s orientation to determine the axes: e.g. for lateral movement use camera’s right vector, for vertical movement use world up (or camera’s up if we allow tilted roll). Alternatively, in free-look, traditional “pan” might not be needed if we implement explicit movement controls (WASD covers panning motions). Regardless, the controller will support a pointer-driven pan for consistency – mapping to translating the camera’s position.
Zoom/Dolly: Zooming the view will be handled by moving the camera toward or away from the pivot (for orbit modes) or moving the camera forward/back (for free-look). In orbit modes, we maintain a distance from the camera to the pivot. Scrolling the mouse wheel or pinching will decrease or increase this distance, respectively, dollying the camera. To achieve zoom-to-cursor behavior, we incorporate a slight pan during the zoom so that the point under the cursor stays fixed on screen[3]. The idea is that instead of always zooming into the center of the view, we zoom towards the area of interest under the mouse. Implementation: when a zoom input occurs, we cast a ray from the camera through the cursor’s position into the scene (if we have a depth buffer or scene geometry, we can find the intersection point near the cursor). If an intersection point is found, we will use that as the anchor. If not (e.g. empty background), we can use the ray at the depth of the current pivot or some default plane. We then compute a new camera position such that after moving, the cursor’s world point is aligned. This can be done by a combination of moving the camera forward along its view direction and panning slightly. Another simpler approach is: project the pivot point into screen coordinates; when zooming, shift the pivot in screen-space toward the cursor by some factor to simulate that anchor. Either way, the effect we achieve is that whatever was under the cursor stays roughly under the cursor (and grows in size) as you zoom in[3]. This is a much more natural experience, as noted in CAD software and Google Maps 3D. We will likely include a setting or automatically enable zoom-to-cursor. If for some reason we want the old behavior (zoom to center), we could disable that calculation.
Zoom Limits & Behavior: We will define reasonable limits for zoom distance to avoid going through the pivot or too far out. For example, stop at a small minimum distance (or switch to a different navigation if attempting to go inside an object) and a far maximum (or when the scene extents are fully visible). If using a perspective camera, zooming is implemented by dolly movement (not by changing the FOV, which remains fixed for predictability). In an orthographic projection case (if any), zooming would adjust the orthographic scale, but the logic of zoom-to-cursor still applies (adjust the center offset accordingly).
Inertia and Smooth Motion
To give a professional polish, the camera controller will support inertial smoothing for rotations, panning, and zooming. This means if a user “flicks” the view (a quick drag and release), the camera will continue moving for a short duration, decelerating to a stop. In practice, we’ll implement this by storing the last input velocities: e.g., as the user drags, we compute the rotation angular velocity (degrees per second or similar) from the recent events. On mouse release, if the velocity exceeds a threshold, we enter an inertia loop where the camera continues to update its rotation each frame using a gradually decreasing velocity. We’ll apply a exponential damping so that it smoothly comes to rest. The same applies to pan: if you release the mouse while panning quickly, continue translating the view a bit. For zoom, if the user uses a touchpad fling or similar, continue dollying in/out briefly. This feature makes the controls feel “weighty” and natural, and helps in examining objects (users can spin the model and let it rotate to see all sides). We will need to integrate this with the rendering loop – likely using requestAnimationFrame to continue updating the camera during inertia until motion stops.

    Progress 2025-11-26: Added quaternion-based arcball inertia (axis + angular speed) plus existing pan/turntable decay, so flicks now continue smoothly regardless of orientation.
We will ensure that inertia is applied only when appropriate (e.g., not when a user intentionally makes a precise small move). Also, if the user engages another interaction (new drag or key press) while inertia is ongoing, we cancel the inertia immediately to retain responsiveness.
The inertia implementation requires keeping some transient state, e.g. a velocity vector or separate angular velocities for yaw and pitch in turntable mode, or an axis-angle velocity in arcball mode. A simple approach for arcball is to use the last few quaternion differences to derive an angular velocity axis and magnitude. We can simplify by treating turntable and arcball somewhat differently: in turntable, track separate velocities for yaw and pitch (and maybe for roll if ever needed), then update angles each frame; in arcball, store an axis and velocity magnitude. Either way, this will be encapsulated in the CameraController’s update loop. The result is a smoother experience, closer to what users expect from modern 3D viewers.
Focus and Pivot Re-Centering
The controller will support double-click (or tap) to focus on a point. When the user double-clicks a location on the model, we perform a raycast in the scene to find the 3D point under the cursor. If a point is found (e.g., on the mesh surface), we will smoothly transition the camera’s pivot to that point. In an orbit mode, this means the new target of the camera becomes that point. We likely also adjust the camera distance if needed so that the point is comfortably in view (for example, if the point was very close, we might back up a bit so the view is not extremely zoomed; or if it’s far, we might move closer). A common implementation is to move the camera such that the clicked point becomes the center of the view. In Blender’s terminology, this is like “View Selected” – the view orbits around the chosen element[4]. We can achieve it by setting: pivot = clickedPoint. Then set camera.position = pivot - forward * distance where distance could remain the same or be adjusted based on scene size. We then update the orientation to look at the new pivot (if we keep the same forward direction, the camera might already be looking at it; if not, we can interpolate the orientation to directly face the pivot to avoid any slight misalignment). This effectively recenters the rotation point to where the user wants to focus.

To keep things smooth, we won’t teleport the camera unless the distance change is minimal. Ideally, we implement a short interpolation (tween) when refocusing: over, say, 200-500ms, move the camera from its old position to the new position, while updating orientation to keep the target in view. This gives a nice zoom/pan toward the point. If implementing an animation is too involved, we can also instantly move it, but ease-in/out animation is a nice touch for UX.
In free-look mode, double-clicking could either be disabled or interpreted differently (since there is no orbit pivot). We have a couple options: we could treat double-click as a “teleport” – move the camera to that location or a fixed offset from it. However, that might confuse the user. Alternatively, in free-look, we might use double-click to look at a point: i.e., rotate the camera so that its forward vector aims at the clicked point (without moving the camera position to it). This would be akin to a quick orientation reset towards a feature of interest. It might be useful if the user gets lost orientation-wise. We can choose the behavior based on the use-case; for a CAD viewer context, orbit modes will likely be the primary mode where double-click refocus is needed.
Additionally, we could provide preset “home” or orthographic views (not requested but often useful): e.g., buttons to quickly align camera to Top/Front/Side view or reset to an isometric. This can be built by setting specific orientations and perhaps pivot = model center. This is outside the direct scope, but our system would make it easy to implement these since we can set camera state programmatically.
Data Structures and State Management
We will introduce some structured types to keep camera state cleanly:
•	CameraMode: an enum or union type for "turntable" | "arcball" | "free". This can be part of the controller’s state so we know which logic to apply.

•	CameraState: an interface or class to hold the camera’s current committed state, potentially like:

 	interface CameraState {
    mode: CameraMode;
    position: vec3;       // camera world position
    orientation: quat;    // camera orientation (rotation) as a quaternion
    pivot?: vec3;         // orbit target point (undefined in free-look mode)
    distance?: number;    // distance from camera to pivot (for orbit modes)
    fov: number;          // field of view for projection (if needed)
    // Additional basis vectors can be derived from orientation as needed
}
 	The state will contain pivot and distance only when in an orbit mode. We keep orientation as a quaternion for robust math; the rendering code (WebGPU shader) might still consume a view matrix or forward/up/right vectors, which we can derive each frame from position and orientation. We will ensure to update the displayCamForward, displayCamUp, etc., from the quaternion: e.g., forward = orientation * (0,0,-1) assuming looking down -Z, etc. This way, the rest of the rendering pipeline remains unchanged, getting a consistent camera basis.
•	Transient State for Interaction: We will not clutter CameraState with temporary variables used only during a drag. Instead, our CameraController will use local variables while processing input, such as:
•	dragStartOrientation (quat) and dragStartPivot (vec3) saved on pointer-down.
•	initialMousePos for reference, and possibly lastMousePos to compute deltas or velocity.
•	currentDragQuat (quat) for arcball to represent the in-progress rotation from drag start to current.
•	velocity (vec3 or separate components) for inertia as discussed.
These are reset when an interaction ends. By segregating them, we keep the committed CameraState clean and only updated at appropriate times. This aligns with having transient vs. committed camera basis: e.g., during a drag, the displayed camera orientation might be q_now = q_drag * q_start but we don’t assign q_now back into the state’s orientation until the drag is finalized (or if we do continuous update, we still have dragStartOrientation to fall back on if needed).
•	Input Configuration: We will define what inputs map to what actions in a clear way, possibly as an config object or constants. For example: LeftMouse = rotate (orbit or free-look rotate), RightMouse (or Shift+Left) = pan, MiddleMouse = alternative pan, Scroll = zoom, DoubleClick = focus, Touch pinch = zoom, two-finger drag = pan, one-finger drag = rotate (or translate in free-look), etc. This mapping could be customizable but will default to industry standards (similar to Blender/Three.js conventions). Having this clearly defined helps ensure consistency. And because the CameraController encapsulates it, webgpu_core.ts just needs to forward events to it with event type and positions.

If free-look, adjust state.position directly along camera axes. - If zooming (wheel): call a dollyBy function that adjusts state.distance (orbit) or moves position (free-look). If implementing zoom-to-cursor, provide the cursor position to that function so it can compute the needed pan-offset.
On handlePointerUp, we conclude the interaction: if it was a rotation drag and inertia is on, compute the final velocity and store it. If not, just end. We also commit the final state: for example, in arcball, we would multiply and store the final quaternion as the new base orientation (if we weren’t directly updating it). In practice, since we likely update state.orientation each frame, it’s already up-to-date; but our stored dragStartOrientation can be discarded. Similarly, clear any transient data.
If double-click, we get the clicked point (via raycast callback from the app or using depth texture if available) and call focusOn(point). The focusOn method will set the new pivot and optionally animate the camera move.
The updateInertia(deltaTime) method will be called from the animation loop or rendering loop. It applies any ongoing inertia: e.g., if rotation inertia is active, update the orientation by small increments (for turntable, increment yaw/pitch by velocitydt; for arcball, slerp quaternion by a small rotation around stored axis). It also applies friction (reduce velocity). Same for pan (move position by velocitydt) and zoom (adjust distance by velocity). When velocities drop below a threshold, it stops. This method will ensure the camera state is smoothly changing even after user input ends, until motion stops. The webgpu_core.ts can call CameraController.updateInertia() on each frame before rendering.

We ensure that renderFrame uses cameraController.state.position and orientation to construct the view matrix. If needed, we update the displayCamForward/up/right global variables (or whatever the existing pipeline uses) each frame from the quaternion. For example:

const { position, orientation } = cameraController.state;
const forward = orientation.rotateVector( (0, 0, -1) );  // assuming -Z forward
const up = orientation.rotateVector( (0, 1, 0) );
const right = orientation.rotateVector( (1, 0, 0) );

and then build a view matrix that looks from position toward position + forward with up as up-direction. This keeps all existing shading or other code that uses the camera basis consistent with the new control scheme.
We will remove any old clamping logic (like if there was clamping of pitch or weird pole-checks) because the quaternion approach no longer needs those. Also, any ad-hoc state like “current yaw angle” or “orbit angle” in webgpu_core.ts will be superseded by the CameraController’s state.
4. User Interface Integration: Ensure there are ways to toggle modes (e.g., a UI checkbox “Arcball Rotation” vs “Turntable Rotation”). This might simply call cameraController.setMode('arcball') or similar. The double-click focus likely needs the application to perform a ray-pick. If the 3D viewer can provide the depth buffer or if we know the geometry, we can integrate a picking function. If not, as a placeholder, we might focus on the point on the near plane (less ideal). Ideally, we integrate with the 3D engine’s picking mechanism.
5. Testing and Tuning: We will test each mode’s feel with real input to adjust sensitivity (rotation speed per pixel, zoom speed scaling, etc.). The “natural drag direction” is ensured by trying the controls: if something feels inverted or non-intuitive, we flip the sign or tweak the axis. We’ll also test the pole behavior: in turntable, verify that crossing the top doesn’t create a sudden flip. The expected behavior is that once the camera goes upside down, dragging left-right appears reversed relative to object (because the camera is inverted). Some tools solve this by switching the reference of “left-right” when inverted; we can consider implementing a logic: if upside down, invert the horizontal drag direction to maintain consistency from the user’s perspective. This can be done by checking the camera’s up vector dot with world up; if negative (camera upside), we flip the yaw input. Such details will be fine-tuned.
6. Documentation & Maintainability: We will document the methods and state clearly in comments (and possibly in a markdown for the team). The design follows PotFoundry’s clean separation ethos: the CameraController can be seen as part of the view-layer logic, while the math in camera_basis.ts is low-level. This separation makes it easier to modify one without risking the other. For example, if in the future we swap out WebGPU for another renderer, we can keep CameraController the same and just adapt how it applies the camera to the new renderer. Or if we want to add a new mode (say an orthographic planar navigation), we can extend the mode enum and add logic without affecting unrelated parts.
In summary, this plan delivers a comprehensive camera control system with turntable, arcball, and free-look capabilities, smooth pan/zoom with zoom-to-cursor, inertial motion, and intuitive pivot focusing. By structuring the solution around a dedicated controller and robust math foundations, we ensure the implementation is maintainable and consistent with existing camera basis usage. This will dramatically improve the 3D viewer’s usability, matching the polish of professional CAD and DCC applications in terms of camera manipulation. Each of the required features will be handled in a modular way, resulting in a fluid and user-friendly camera control in the Streamlit WebGPU preview.
